# SMT 歷史報表與統一訂單識別設計

日期：2026-07-22  
狀態：產品負責人已於對話確認整合  
基準：`order-v1-30`／遠端提交 `ba44d43`

## 目標

將適合店內即時營運的舊 POS 報表能力加入 SMT，並一次修正報表只能看當日、付款方式不能逐項對數、各頁訂單流水不一致三個問題。商品結構、權限、供應商、成本及報表設定仍由 Admin 管理，不在本輪加入 SMT。

## 範圍

### SMT 本輪加入

- 日期範圍：今日、昨日、最近 7 日、最近 30 日、最近 3 個月、最近 6 個月、自訂日期。
- 分析分頁：營業、渠道、付款、商品、異常。
- 營業：原額、淨實收、有效訂單、平均客單價、折扣、退款、未結帳、每小時趨勢及日結紀錄。
- 渠道：現場、堂食、網站／App、電話／WhatsApp、Keeta、Foodpanda 等，顯示單數、原額、折扣、應收、實收、退款及平均客單價。
- 付款：現金、Alipay、WeChat Pay、PayMe、FPS、組合付款、平台代收及待核實，顯示單數、應收、實收、退款、差額及對數狀態。
- 商品：產品排行及分類排行，顯示數量、金額及佔比。
- 異常：取消、退款、付款待核實、打印失敗及敏感操作摘要。
- 每個渠道及付款項目可打開對應訂單清單。
- CSV 匯出跟隨目前日期範圍。

### 留給 Admin

- 商品、套餐、價格、分類、口味及內部簡稱設定。
- 付款戶口、權限、成本、利潤、銷售目標、供應商及採購管理。
- 報表模板、跨店匯出、歷史資料更正及完整稽核管理。

## 訂單識別

每張新訂單保存兩組識別：

- `displayOrderNo`：店內每日叫號，格式 `P001` 至 `P999`，以早上 05:00 為營業日分界，所有渠道共用同一序列。
- `id`：永久唯一訂單編號，包含營業日、時間、毫秒、終端及每日序號；堂食另保存 `dineTableId`，不把可變資料只塞入編號字串。

舊訂單只有 `id: P0055` 時繼續可讀；所有畫面用 `displayOrderNo || legacy id` 顯示。新訂單關聯、打印、稽核及更正使用永久 `id`。超過 `P999` 時禁止建立新單並顯示明確錯誤，不循環覆蓋。

實作邊界：`order-v1-31` 可在同一本機同步歷史及活躍堂食中統一派號；兩部完全獨立實體終端同時離線建單仍須 Firebase／後台原子遞增 API，未接通前不列為跨機完成。

## 資料流

1. 結帳或堂食完成前，由共用識別模組讀取完整歷史訂單。
2. 模組按 05:00 營業日篩選當日訂單，找出最高 `dailySequence` 或舊 `Pxxx` 顯示號碼。
3. 產生下一個 `displayOrderNo`、`dailySequence`、`businessDate` 及永久 `id`。
4. 各主頁由同一模組按訂單實際時間找最新顯示號碼。
5. 報表頁把選定日期範圍轉成明確 `start`／`end`，由訂單明細重算全部摘要及分拆，不保存易失真的每日總數。

## 相容及錯誤處理

- 保留現有 localStorage key 及舊訂單結構。
- 缺少付款、產品分類或時間的舊資料放入「未分類」，不令報表白屏。
- 自訂日期結束早於開始、超過六個月或格式無效時拒絕套用並顯示原因。
- 今日未有訂單時所有分頁顯示零值及空狀態。
- 報表只讀訂單，不直接修改歷史；更正仍走既有有原因及 audit 流程。

- 報表測試覆蓋全部日期預設、自訂範圍、付款分拆、渠道分拆、退款、未結帳、產品分類、時段及訂單下鑽索引。
