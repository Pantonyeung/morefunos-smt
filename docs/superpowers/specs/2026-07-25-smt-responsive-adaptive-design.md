# SMT Responsive Adaptive Design

## 目標

以已驗證可運行的 1920×1080 SMT 基準為唯一乾淨來源，建立一套多尺寸自適應架構；未來修改 UI 時，所有受支援橫屏尺寸同步適配，不再為 1920 或 1280 建立獨立補丁版本。

## 基準與分支保護

- 永久乾淨基準分支：`smt-responsive-from-1920-baseline`
- 基準 commit：`5d7f9726305a0c9aad7b6238e776d4a7ac670009`
- 自適應工作分支：`smt-responsive-adaptive-v1`
- 禁止直接修改乾淨基準分支。
- 禁止將 QA 測試檔案、臨時 override、單尺寸修正寫入乾淨基準分支。

## 自適應原則

1. 所有 SMT 頁面共用同一套 responsive 規則。
2. 不再以固定 1920×1080 或 1280×800 畫布作為唯一渲染方式。
3. 主要依據實際可用 viewport 寬、高及比例決定 layout。
4. 1920×1080 為 Primary 驗收尺寸；1280×800 為 Secondary 驗收尺寸。
5. 同時驗收：1600×900、1440×900、1366×768。
6. 同一功能不得因尺寸不同而有兩套商業邏輯或兩套 DOM 結構。

## 中央尺寸系統

集中管理以下 UI token：

- 購物車／產品區比例
- 產品卡欄數
- 產品卡高度與圖片比例
- 字體大小
- gap／padding
- category 區高度
- 快捷飲品區尺寸
- modal／popover 最大寬高
- 頂欄及底部操作區高度

各頁面只能讀取中央 token，不可自行另寫單尺寸 override。

## 建議版型層級

- Large：1600px 或以上
- Standard：1200–1599px
- Compact：1024–1199px

實際 breakpoint 以視覺與操作驗收為準，不以裝置型號名稱硬編碼。

## 禁止事項

- 禁止新增 `t2s-1280.css` 類型的單機型補丁層。
- 禁止由 shell 動態 inject page CSS 修正子頁。
- 禁止大量新增 `!important` 作為主要 layout 控制方式。
- 禁止同一元件在 page.css、HTML inline style、loader 三層重複定義尺寸。
- 禁止為單一尺寸複製一套 page.js 或商業邏輯。

## 兼容策略

現有 1920 基準的：

- 點單流程
- 購物車
- 套餐／飲品邏輯
- 掛單／取單
- 待處理
- 結帳／訂單／堂食／售罄／更多頁面

全部保持功能語意一致。自適應改造只改 layout 與尺寸協調，不主動重寫商業邏輯。

## 驗收標準

每個主要階段完成後必須跑瀏覽器自動化測試，包括：

- 1920×1080
- 1600×900
- 1440×900
- 1366×768
- 1280×800

測試至少覆蓋：

- 頁面載入
- 快速產品操作
- 購物車加減
- modal 開關
- 頁面切換
- viewport 反覆切換
- 重載後狀態穩定性

任何尺寸出現白屏、主要操作不可點擊、modal 超出可操作範圍、購物車或產品區不可用，均視為失敗。

## 已知基準問題

部分舊產品圖片資源路徑存在 404。此問題不影響目前核心操作，但需要獨立修復；不得用 responsive 改造掩蓋或混合處理。

## 成功定義

完成後只保留一套 SMT UI 結構與功能邏輯。之後修改共用元件或中央 UI token，所有受支援尺寸應同步取得一致更新；如需要尺寸差異，只能透過中央 responsive 規則處理，不可建立獨立補丁分支或獨立頁面版本。
