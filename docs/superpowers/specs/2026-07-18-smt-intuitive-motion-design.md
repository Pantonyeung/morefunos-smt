# More Fun／磨飯 SMT｜輕日系直覺操作與動效系統設計規格

版本：V0.2 Design Spec  
日期：2026-07-18  
適用範圍：SMT-00 至 SMT-16；第一實作切片聚焦 SMT-00 至 SMT-05  
權威依據：`smtlock.zip` 內的 SMT Master UI／Logic Lock、Slide Master Lock、Roadmap 與 Web Core

---

## 1. 設計目標

把目前 SMT Web Core 從「可展示的介面骨架」推進為「第一次使用也能自然完成點單的前線操作介面」。

本輪成功不以動畫數量或視覺華麗程度衡量，而以以下結果衡量：

1. 新員工不需要背操作路徑，畫面本身能提示下一步。
2. 高峰時主要操作不需要四處尋找，常用按鍵位置穩定。
3. 每次觸控都有即時、安靜、明確的結果回饋。
4. 付款、打印、同步、訂單時間等狀態互不混淆。
5. 錯誤、斷網或打印失敗不清空工作內容，不造成廣泛鎖機。
6. 所有視覺與動效符合磨飯的輕日系、現代、簡約及溫暖品牌氣質。

核心判斷句：

> 不要求使用者記住系統；系統要在每一步告訴使用者現在在哪裡、剛才發生了甚麼、下一步可以做甚麼。

## 2. Lock 邊界

### 2.1 絕不能改動

- 五項主導航及順序：點單｜訂單｜堂食｜售罄｜更多。
- SMT 本機為正式 Order、流水、付款、Print Job、Day Close 權威。
- Required 未完成才是點單主要阻塞之一；一般備註、電話、稱呼、外部參考號不得阻塞。
- 接受或「先製作」後才建立正式訂單、流水、Print Job 及啟動 30 分鐘。
- 堂食 30 分鐘 Order Timer 與 35 分鐘 Table Session 提示雙軌獨立。
- 暫存不派流水、不打印、不啟動 30 分鐘。
- 「稍後處理」不代表接單、拒單或停止待處理提醒。
- 售罄、打印、同步、備份、Token、API 或更新異常不得廣泛鎖死本機 POS。
- 大型單、預訂單、員工 PIN、主管審批、遠端鎖機與強制更新不得重新加入。

### 2.2 可以優化

- 資訊層級、字級、間距、觸控面積、對齊、視覺重量。
- 按下、選取、加入、刪除、完成、警告、恢復等互動回饋。
- Drawer、Modal、Toast、角標、狀態卡及頁面切換動效。
- 將低頻資料收進狀態中心或二級面板，保持主要工作區清晰。
- 空白、錯誤、離線、重試、恢復及撤銷流程。
- 快速模式的資訊密度與動畫節奏，但不得改變業務規則。

## 3. 三種方向比較

### 方案 A｜只做視覺美化

保留現有結構，只更換色彩、圓角、陰影和動畫。

優點：最快。  
缺點：搜尋、付款選擇、產品選項、狀態混合等原有操作問題仍在；看起來更新，但不會真正更易用。

### 方案 B｜互動優先的漸進揭示（採用）

保留 Lock 結構及工作區比例，重新建立元件狀態、按鍵層級、固定操作位置、Drawer 流程與動效語言。畫面先顯示必需資訊，詳細內容按需要展開。

優點：學習成本最低；適合夫妻店、高峰操作及平板觸控；能逐 Slide 實作，風險可控。  
缺點：需要先整理狀態模型及共用元件，初期工作量高於純美化。

### 方案 C｜高品牌感沉浸式動效

大量使用轉場、粒子、插畫和品牌敘事。

優點：展示感強。  
缺點：會分散 POS 注意力、增加 GPU／電量負擔，也容易令新員工找不到操作入口，不適合 SMT 日常營運。

**正式選擇：方案 B。** 品牌動效只在成功、恢復及空白狀態小量使用，不作持續背景裝飾。

## 4. 設計語言：輕日系現代簡約

### 4.1 視覺性格

- 大面積米白與暖灰，保留呼吸感。
- 文字深啡黑，不用純黑壓迫畫面。
- 橙色作主要行動與磨飯識別色，不把整個介面染成橙色。
- 卡片邊界以細線及微弱層次區分，避免厚重陰影。
- 圖示簡單、線性、可辨認；所有高頻核心功能同時保留文字標籤。
- 食物圖是決策資訊，不是裝飾；圖片、名稱、代碼、價格必須形成固定閱讀順序。

### 4.2 六套主題的處理

保留 Lock 中六套主題，但改為「共同中性底＋季節性強調色」：

- 所有主題共用相同背景、字色、Success／Warning／Danger／Offline 語義。
- 只更換主色、淡色選中底及少量插畫點綴。
- 主題切換不改按鈕位置、狀態含義或元件尺寸。
- 暖橙為預設及 Figma 主樣板。

### 4.3 基礎 Token

- 最小高頻觸控面積：一般模式 48×48；快速模式 56×56。
- 基礎圓角：8／12／16；不混用過多半徑。
- 基礎間距：4／8／12／16／24／32。
- 正文建議 16–18px；關鍵價格、流水、倒數 24–32px。
- 主要按鈕永遠使用實心主色；次要按鈕使用白底描邊；危險操作不用主色。
- 所有重要狀態同時提供圖示、文字及色彩，不靠顏色單獨傳達。

### 4.4 繁體中文與強閱讀性硬鎖

- 所有操作介面、按鈕、提示、狀態、錯誤及空白畫面一律使用繁體中文；不得混入簡體中文。
- 技術詞只可在必要時出現，並優先轉成店舖人員看得懂的營運語言。例如顯示「離線可工作」，而不是只顯示 `offline mode`。
- 高頻按鈕不得只顯示英文、縮寫或圖標；必須同時提供清楚的繁體中文文字。
- 正文不得小於 16px；次要說明不得小於 14px；金額、流水、倒數及待處理數量使用 20–32px。
- 文字與背景至少達到 WCAG AA 對比；核心操作與關鍵狀態以更高對比為目標。
- 圖標採用一致的線性圖標系統，正常尺寸不小於 22px；高頻導航及狀態圖標以 24–28px 為主。
- 不使用難以辨認的 Unicode 符號代替正式圖標；圖標旁保留文字，避免員工猜意思。
- 商品圖片需使用固定比例、清楚裁切及失圖替代狀態；圖片不能壓過商品名稱、代碼及價格。
- 售罄、暫停、付款待核、打印異常及離線狀態必須同時使用圖標、短文字和語義色。
- 「簡約」只代表減少雜訊，不代表縮小文字、淡化重要資訊或隱藏主要操作。

## 5. 直覺操作原則

### 5.1 看得見，不靠記憶

- 高頻功能不可只藏在長按、右鍵、滑動或「⋯」內。
- 「⋯」只容納低頻設定，不承擔唯一入口。
- 核心按鈕使用動詞：加入、修改、暫存、取單、結帳、確認落單、稍後處理。
- 每頁只設一個視覺最強的主要行動。

### 5.2 位置穩定

- 主要確認鍵固定在 Drawer／Modal 右下或底部固定區。
- 返回／取消固定在主要確認鍵左側。
- 購物車 Summary、暫存／取單與結帳位置不因商品數量跳動。
- 新資料到達時不重新排序使用者正在觸控的卡片；以角標、插入動畫或待處理區更新。

### 5.3 直接操作

- 按商品卡直接加入；需要 Required 時自然進入整理面板。
- 購物車商品行直接加減；按商品行進入該 CartItem 修改。
- 售罄與暫停供應可在原位置切換，狀態改變後位置不移動。
- 訂單、枱卡及打印項目按卡片進入詳情，不要求先選取再找操作列。

### 5.4 容錯及可逆

- 清空購物車、拒絕新單、取消 Print Job、清枱、恢復資料等高風險操作必須二次確認。
- 單項移除、售罄切換、暫存等低風險操作優先提供 5 秒撤銷，不使用阻塞式確認。
- 按鈕連按由 Idempotency／本機交易防重，不以禁用整個畫面處理。
- 任何保存失敗先保留畫面資料，再顯示可重試原因。

## 6. 全局框架與狀態中心（SMT-00／15）

### 6.1 頂部狀態列

分成三個閱讀區，不讓每個狀態都同樣搶眼：

1. 左：品牌、營業狀態、離線可工作／系統在線。
2. 中：當前流水、新單、付款待核、打印異常等真正待處理項目。
3. 右：營業日、時間、快速模式、主題及狀態中心入口。

正常狀態保持安靜；只有需要人處理的項目出現實心角標或短暫脈衝。

### 6.2 全局狀態中心

以「現在要不要處理」排序，而不是按技術名稱排序：

- 立即處理：本機不能安全保存、付款需核實、供應衝突。
- 可稍後：打印重試、同步積壓、備份失敗。
- 正常資訊：Catalog 版本、最近備份、裝置連線。

每一項顯示：狀態名稱、影響範圍、最後更新、建議行動。不得只顯示紅點或技術錯誤碼。

### 6.3 底部導航

- 五項位置固定，不因頁面或角標改變寬度。
- 圖示在上、文字在下，Active 狀態以底色＋短指示條表達。
- 角標只顯示真正待處理數量；超過 99 顯示 `99+`。
- 切頁採 140–180ms 淡入／輕位移，不使用大幅滑屏。

## 7. 第一垂直切片的操作設計

## 7.1 SMT-01 啟動

- 品牌 Loading 只顯示 0.6–1.2 秒，資料就緒立即進入上次工作頁。
- 背景檢查以頂部狀態逐步恢復，不用阻塞式「正在檢查 1/8」。
- 只有本機資料無法安全開啟時進修復頁。
- 離線啟動顯示一次「離線可工作」，2 秒後收成頂部狀態。

## 7.2 SMT-02 點單主頁

### 版面

- 左側 30%：Order Workspace；右側 70%：分類、搜尋、商品 Grid、快捷飲品。
- 左側 Summary 及底部操作固定；商品行獨立捲動。
- 分類保持可掃視，搜尋作最後固定入口；打開後不令商品區大幅跳位。

### 商品卡

固定閱讀順序：代碼 → 名稱 → 價格 → 圖片 → 狀態／數量。

狀態：

- Default：白底細框。
- Pressed：縮小至 0.98，80–100ms。
- Added：卡片主色描邊閃現一次，數量角標增長，購物車對應行短暫高亮。
- Required：加入後直接開「先整理」面板，明確顯示尚欠項目。
- Sold out／Paused：保留位置、降低圖片對比、顯示完整文字狀態，不可觸發加入。

### 購物車

- 新商品在對應位置淡入，不讓整個列表重繪閃動。
- 數量加減只更新該行及 Summary。
- 修改商品保留分類及商品捲動位置。
- Required 未完成：主按鈕顯示「先整理 · 尚欠 N 項」。
- Required 完成：主按鈕顯示「結帳 $金額」。
- 空白：主按鈕 disabled，文案「先由右邊選擇餐點」。

## 7.3 SMT-03 產品詳細／Required／Pool／Link Up

- 使用右側 Drawer／工作面板，左側購物車保持可見。
- 頂部固定商品名、目前價錢與完成度；底部固定取消／確認。
- Required 區永遠排在最前；完成後自動收合但可重新打開。
- 每一區顯示「選 1」「尚欠 2」「已完成」，避免使用 technical key。
- 無效或售罄選項保留原位置並說明為何要重選。
- 修改現有 CartItem 只更新該項，不重建整個購物車。

## 7.4 SMT-04 結帳

- 保留左右雙欄，但右側改成由上而下的自然決策順序：來源 → 模式 → 包裝 → 優惠 → 付款 → 可選資料 → 確認。
- 選中付款方式必須有勾選圖示及文字，不只靠背景色。
- 備註、電話、稱呼、外部參考號歸入「其他資料（可選）」摺疊區，避免畫面一開始過長。
- 總額、盒費、優惠分行顯示；總額固定靠近確認鍵。
- 正式確認期間按鈕顯示「正在安全保存…」，其餘畫面保留；成功後才清購物車。
- 保存失敗：按鈕恢復、資料不清空，顯示原因及「重試」。

## 7.5 SMT-05 新 App／Web 訂單

- 全局彈窗只在新批次第一次到達出現；多張合併成一個批次摘要。
- 聲音／震動只在有新批次時一次；未處理以每分鐘批次提醒，不連續逐張阻塞。
- 彈窗固定兩個行動：「稍後處理」「查看及處理」。
- 「稍後處理」後頂部提示與訂單角標保留，並以細小回收動畫指向訂單導航。
- 快速核對層將供應衝突、付款待核及重要備註放在商品清單上方。
- 接受／先製作成功後才出現流水及 30 分鐘起點。

## 8. Motion System

### 8.1 原則

- 動效回答三件事：按到了嗎、改了甚麼、結果去哪裡。
- 不使用無意義背景漂浮、循環發光或大幅彈跳。
- POS 高峰保持 60fps；主要使用 opacity 與 transform。
- 卡片在使用者按住或編輯時不自動跳位。

### 8.2 時間 Token

- Touch feedback：70–100ms。
- 選中／角標／數字更新：120–160ms。
- 頁面小轉場：140–180ms。
- Drawer／Bottom sheet：220–260ms。
- Modal：200–240ms。
- Toast 進入：160–200ms；停留 1.6–2.4 秒；離開 120–160ms。
- 警告脈衝：只執行 1–2 次，不無限循環；未處理狀態以固定角標保留。

### 8.3 具體動效

- 商品加入：卡片壓感 → 數量角標更新 → 購物車對應行淡入／高亮。
- 總額更新：數字短距離切換，不使用跳動 Count-up 長動畫。
- Drawer：由右側進入，背景只加輕遮罩，不重度模糊。
- Toast：由底部安全區上升，文字說明結果；提供撤銷時保留按鈕。
- 新單：彈窗出現＋一次脈衝；稍後處理後縮入訂單角標。
- 成功保存：確認鍵變為完成狀態 300–500ms，再切換頁面。
- 磨飯品牌記憶種子：只可在日結成功、備份完成、首次裝置設定等低頻成功節點輕量出現；高頻加商品不使用粒子。

### 8.4 快速模式

- 保留所有必要觸控回饋，但將 Drawer、Modal、頁面轉場壓縮約 25–35%。
- 取消非必要品牌點綴、圖片浮現及延遲。
- 增大高頻按鍵與商品名稱，不用「更密」換取速度。

### 8.5 Reduced Motion

系統或使用者開啟減少動態時：

- 位移轉為短淡入。
- 取消縮放彈性、脈衝及品牌粒子。
- 保留文字、角標、勾選和 Toast 回饋。

## 9. 即時狀態與資料視覺化

SMT 是營運工作區，不做等權重 KPI 卡片牆。

- 首屏只顯示當下需要決策的狀態。
- 訂單倒數以剩餘時間＋明確文字呈現，不依賴圓環或顏色。
- 25–29 分鐘維持正常；最後 5 分鐘使用 Warning；到點移入歷史，但付款／打印異常標籤仍保留。
- 堂食 35 分鐘顯示一次提示與持續角標，不重複閃動。
- 打印機狀態使用「可用／重試中／離線／改送」及最後成功時間，不顯示難懂的連線圖表。
- 報表頁的圖表另於 SMT-13 規格中處理，不把分析圖塞入日常點單頁。

## 10. Accessibility 與實機使用

- 文字與背景達到可閱讀對比；狀態不只靠色彩。
- 所有互動元件有可見 focus 狀態，支援外接鍵盤及掃碼設備。
- Icon-only 按鈕必須有可讀標籤及足夠觸控區。
- 不以 hover 作主要資訊入口。
- 彈窗打開後焦點進入彈窗；關閉後回到原觸發位置。
- 支援 1024px 級橫屏及 Sunmi T2s；鍵盤彈出時主要確認鍵仍可見。
- 斷網、低效能及背景恢復時保留最後有效畫面，不顯示空白頁。

## 11. 現有 Web Core 的實作風險

以下問題需在實作階段處理，不可只靠 CSS 美化：

- `currentNo()` 固定返回 `P0056`。
- 搜尋入口把 query 設為空格，可能導致錯誤空結果。
- 付款按鈕只呈現外觀，未保存真正選擇。
- 產品選項與備註未寫入 CartItem。
- 新單「查看及處理」未形成完整接受／先製作／付款問題流程。
- 目前以 `localStorage` 模擬資料，不是正式本機交易及 Idempotency。
- 訂單、打印、報表及狀態資料大量硬編碼。
- 每次 `render()` 重建整個 App，會造成焦點、捲動及觸控中的卡片跳動風險。
- Service Worker cache 版本及更新回退策略未達 Lock 要求。

實作必須逐步把「整頁重新渲染」改成狀態驅動的局部更新或明確元件邊界，但不做與本輪無關的大型重構。

---

## 12. 元件邊界

第一階段建立以下可獨立理解及測試的元件：

- `AppShell`：安全區、TopStatusBar、Workspace、BottomNavigation。
- `TopStatusBar`：只負責摘要及狀態中心入口。
- `StatusCenter`：狀態分級、影響、行動及更新時間。
- `ProductCard`：商品展示、供應狀態、加入回饋。
- `CartItemRow`：數量、選項摘要、Required、修改與移除。
- `CartSummary`：件數、盒費、優惠、總額與主行動。
- `ProductRulePanel`：Required、Pool、Link Up、備註及完成度。
- `CheckoutPanel`：來源、模式、包裝、優惠、付款及安全保存。
- `NewOrderBatchModal`：批次提醒與稍後處理。
- `ToastUndo`：非阻塞結果及撤銷。
- `MotionProvider`：正常、快速、Reduced Motion 三套節奏。

元件只透過明確事件及資料模型溝通，不直接修改其他元件內部狀態。

## 13. 測試與驗收

每個第一切片畫面至少驗證：正常、空白、錯誤、離線、恢復。

關鍵測試：

1. 第一次使用者可在不看說明下完成：加入商品 → Required → 結帳 → 付款 → 正式落單。
2. 點擊商品後 100ms 內出現觸控回饋，300ms 內看見加入結果。
3. 返回產品主頁保留分類及捲動位置。
4. 修改 CartItem 不影響其他商品。
5. Required 未完成時不能正式結帳，但可以暫存。
6. 一般備註留空仍能成單及打印。
7. 本機保存失敗後購物車、付款及備註仍存在。
8. 新單稍後處理後角標及每分鐘批次提醒仍保留。
9. 相同 Submission 重送不重複成單；不同明確提交可成為兩張合法訂單。
10. 快速模式不改業務結果，只改尺寸、密度與動效節奏。
11. Reduced Motion 下沒有脈衝／粒子，但所有狀態仍清楚。
12. 1024px 橫屏不出現核心按鍵被遮擋或需要精細點擊。

## 14. 推進順序

### Phase A｜設計基礎

- 建立 Figma Design System：Token、按鈕、商品卡、購物車行、狀態、Drawer、Modal、Toast、導航。
- 建立暖橙主題及六主題共用語義。

### Phase B｜第一垂直切片

- SMT-00 Shell。
- SMT-02 點單主頁。
- SMT-03 Product Rule Panel。
- SMT-04 Checkout。
- SMT-05 New Order Batch。

### Phase C｜Web Core 實作

- 先在獨立分支製作，不直接破壞現有可回滾版本。
- 每一張 Slide 分開提交及驗收。
- 使用 Mock／本機資料完成完整操作，再接正式本機資料層。

### Phase D｜互動及動效 QA

- 平板橫屏實測。
- 高峰快速模式實測。
- 低效能、斷網、背景恢復及打印異常實測。
- 以錄屏／Remotion 交互演示整理正常、快速及錯誤路徑，作驗收溝通，不把 Remotion 動畫直接塞進 POS Runtime。

### Phase E｜後續 Slide

依 Lock 逐步推進訂單、堂食、售罄、打印、售後、日結、報表、更多與 APK Gate。
